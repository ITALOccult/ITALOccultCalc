AOTA Integration, Binning, and Normalisation

AOTA provides the facility to bin observations to improve the signal-to-noise ratio. Binning is also required when the camera is an integrating camera, so that each set of integrated frames is treated as a single measurement.
  • This step occurs automatically whenever a CSV file is read.
  • This step can be called at any stage during subsequent analysis - by going to the first tab and clicking the Integration/Binning/Normalisation button.
  • For a signal from an integrating camera, this allows the first frame of an integration block to be specified, and for setting the number of frames in an integration block
  • Can be used with a non-integrated signal, to bin the observed frames - again specifying start, plus number of frames to bin.

In principle, for a light curve with potentially multiple events, the binning levels can be set differently for each possible event. If the binning level is changed after measuring an event, and subsequently that event is re-analysed from Tab 4, the final results for that event will be based on the current binning level.

The Point Length setting changes the length of the line for each frame displayed. The smaller the value, the greater the number of points that are displayed.
  • The set of data points that are used for the first binned data point is indicated with a yellow rectangle. There are two ways of setting the location of the rectangle:

For a signal from an integrating camera, match the rectangle to a set of data points corresponding to a single block of identical output frames. Expected magnitude drop

When analysing a light curve for the presence of an asteroidal occultation, it is useful to have the expected magnitude drop available early in the process - to assist with detection &/or assessment of whether a light drop is consistent with an occultation event. Set the expected magnitude drop in the blue box 2. Expected Mag drop. The value set here will propagate through to other forms used in the analysis. A default Mag drop of 1.0 is set.

If you are simply reporting a light curve, there is no need to set a magnitude drop - as it is not used in that process.

Integration/binning

The form is used as follows:

  • display an appropriate number of data points to see the nature of the light curve, using the 'Display xxx data points'. Up to 4000 data points from the start of the file can be displayed.
  • set the first frame to use in the binning by either positioning the mouse at the relevant point (by left-clicking), or adjusting the 'First frame' in the 3. Binning/integration box at the bottom of the form. A yellow bar will be drawn at this location
  • set the number of frames to be binned by either positioning the cursor at a relevant point and right-clicking, or set the 'Number of frames' under 3. Binning/integration box at the bottom of the form This will result in a yellow box extending over the first group of frames to be binned.

Specific points to note are:

  • If the light curve involves an integrating camera, ensure the yellow box coincides with a complete group of points corresponding to an integration interval
  • the binning can be set to multiple integrations - just make sure that the box covers complete sets of integration intervals
  • the width of the box (in frames) should correspond to the numerical value of the integration. For most cameras this will be one of 2, 4, 8, 16, 32, 64, 128 or 256. For some cameras the integration interval may include 6, 12, 24, 48 and 96,
  • if you are unsure where the first full integration interval starts, set the yellow box to an integration interval that is well-defined. You can then re-position the box by merely clicking the left mouse button at the appropriate location - which will be a whole number of integration intervals earlier.

For a non-integrating camera, the usual first frame to use is the first frame! If the SN is good, just 'bin' at the one-frame level (i.e. no binning!). But if you want to improve the SN, increase the number of frames that are binned until you get a light curve that clearly shows the event (if any).

Click OK when done. Clicking cancel will return to the main analysis with no change in binning (and no need to reprocess).

Normalisation

Variable atmospheric transparency (such as light clouds) can have the effect of varying the light levels along the light curve. Such variations are usually present in all the stars in the field of view. If comparison stars are measured, the brightness of the occulted star can be adjusted in accordance with variations in brightness of the comparison star.

Normalisation is achieved by selecting the comparison star to use (using the radio buttons along the bottom of the form), and specifying the number of data points to be used in a running average of the light level of the comparison star. The options available are 5, 9, 15 and 25, with 9 being the default value. The normalisation is performed on the binned values. For example, a running average of 5 is a running average of 5 sets of binned data points (not 5 individual data points)

It is important to note that Normalization is not a process that should be routinely performed. If there are significant long-term variations in the brightness of the occulted star, and a comparison star shows similar long-term variations, then normalisation can correct the long term light variations in the occulted star. However there is always a risk of introducing other problems into the signal.

The following 4 plots illustrate these issues. They are of a 'grazing' occultation, with two separate events being recorded. The observing conditions were not good - especially during the 2nd event, where there are a large number of 'invalid' measurements (marked with blue lines on the horizontal axis) caused by wind-induced vibrations.

The first plot shows the light curve of the occulted star. Note that the bottom of the light curve in the two events is at a different level. Also note that between the two events the light level appears to drop by a small amount.

This second plot has overlaid the light variations of a comparison star. That star appears to show a drop in brightness between the two events. Also during the 2nd event the plot shows numerous gaps - again, caused by invalid measurements.

This third plot has been 'normalised' on the basis of the comparison star, using a 9-point running average. This shows that the light level of the comparison star is more uniform than previous - except in the region of the 2nd event. The problem here is that the use of a running average does not adequately allow for the presence of invalid measurements - leading to distortions in the light curve of both the comparison star, and the occulted star.

This final plot shows the normalised light curve of the occulted star. Note the increased scatter in the regions of the invalid measurements. Ignoring those effects, it may be seen that the light level between the two events now appears to be constant. Also the brightness drop in the two events appears to be fairly similar.

Background

This option is currently only available for light curves measured with Tangra.

The usual method for measuring a light curve involves measuring the light level in a measurement aperture, and a corresponding light level in an equivalent background region - and subtract the background from the signal.  This ensure the signal being measured is based on the star, and not the star and background light.

The problem with doing this is that noise in the background signal is ADDED to the noise in the aperture signal. That is, this subtraction increases the noise in the measured signal.

The 5. Background option provide a choice between:
* subtracting the background on a frame-by-frame basis (the traditional approach); or
* subtracting the overall mean value of the background signal from each frame

This choice is currently only available for videos measured using Tangra. Also when Tangra is used, the option will default to Mean.

The potential benefit of this option is illustrated in the following two plots of the same light curve. The first used traditional point-by-point background subtraction. The second subtracted the mean value of the background noise from each point resulting in less noise overall - with the shallow event being much clearer, and therefore more reliably measurable.
