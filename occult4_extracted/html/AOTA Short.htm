Hints and Tips

Short events - check

This functionality is for assessing whether a light curve has a plausible occultation event lasting just 1 or 2 frames. It may also assist with 3 frame events.

Purpose

This function is designed to check whether there are any very short duration events in the light curve, that might reasonably be caused by an occultation, but which cannot be definitely attributable to an occultation because the light curve is too short to exhibit the classical criterion of having a flat bottom. That is, events of 1 or 2 frame durations, and for 3-frame events where there is doubt the existence of a flat bottom.

This check is not a determination of whether such an event is definitely an occultation. It is not equivalent to the 'Noise induced event' assessment of PyOTE. It is not a general test for whether an event (generally) might be caused by noise. Rather it is simply a check of whether, in the absence of events spanning multiple exposures (where the flat bottom test can be applied), an excursion in a light curve can be reasonably attributed to an occultation event.

The check indicates whether the excursion, is 'probably' or 'possibly' caused by an occultation. When the excursion falls below the 'possibly' level, that excursion is not distinguishable from noise (whether or not there was in fact an occultation).

Subject to any other issues with the observation that may be relevant:

  • a Probable event will be routinely treated as being an occultation event (even though the fact of it being an occultation is not 100% certain).
  • a Possible event will be routinely entered under the new Quality of Fit setting of Short duration event. No astrometry. However Astrometry will not be routinely reported for such events.
  • events that do not meet these two criteria are indistinguishable from the noise, with the presence of an occultation being indeterminate. Such events will be included in the data set of observations as an occultation if (and only if), the observer provides a written explanation of why they believe the light curve indicates the presence of an event. This explanation will become part of their observation, for future reference by those using the data.

In all instances, the observer must provide a .csv file of the measurements to be used for assessment, and to create a light curve file to be archived at VizieR. Without a .csv file, such short events will not be included in the events data base.

 

Basis of the check

The check is based on the following proposition:

When the light curve is smoothed to remove long-term variations:

  1.  A very short event can only be distinguished from the noise if its excursion is significantly greater than all other excursions
  2.  The light curves of all comparison stars can be assumed to not involve any occultation events. Similarly they can be taken to have somewhat similar noise characteristics to the light curve of the target stars. Accordingly, when an excursion is found in the light curve of the target star, there should be no similarly large excursions at any point in the light curves of any of the comparison stars.
  3. The length of the light curve must be long enough to exhibit a full range of noise variations.

Proposition 2 is very important, on the basis that if an excursion similar to that in the target star is found in a comparison star light curve, such an excursion is properly attributable to noise. That is, full confidence in the existence of a non-noise-related excursion in the target star light curve requires a comparison with the light curves of one or more comparison stars. However, noise characteristics/distribution of the comparison brightness is likely to be different from the target star if there is a large difference in their brightness, such that it may not provide a reliable comparison.

Proposition 3 is important for ensuring the noise characteristics of the light curve are well represented. For integrated light curves, the length is based on the number of points after the data has been binned to match the integration.

The results of the test include weighting based on the number of comparison stars used, and the length of the light curve.

Methodology

For each light curve:

  • To reduce the effects of long-term noise variations, a running average of the light curve is generated. 7 possible values for the running value length are provided, with the default value being 25.

    • Large values of the running average reduce the effect of long-term variations, with minimal effect on for mid-term and short-term variations. Small values reduce the effect of both mid-term  & long term variations. Use relatively short running averages only when the signal has clear periodic short-term variations.

  • The light level of the star is then generated as a distance from that running average, for each point along the light curve
  • Compute the Mean, and Standard Deviation, of these distances.
    • IMPORTANT. The method uses the mathematics of a standard deviation provide a convenient basis to compare the data - both within a light curve, and between light curves. The numeric values derived using this approach are not used (and should not be used) to assess a formal probability of the result.
  • Convert the distances to multiples of the standard deviation value. This provides a modified distance from the running average light curve that is based on the distribution of light curve values rather than the distance value.
  • Sort the modified distances by value, with the distance furthest below the light curve having the largest value (distances above the light curve are not used)
  • Repeat for each comparison star. This provides data for each comparison star that may be directly compared with the target star.
  • Apply the Evaluation criteria - see below.

To assist with the assessment of the results:

  • The distribution of the modified distances, for all values greater than 2.0 is plotted for the target, and all comparison stars. Similarity in appearance is an indication of reliability of the comparisons stars for the evaluation. The values are binned in intervals of 0.1 units.
  • The horizontal scale for all plots is set by the largest modified distance from the target and  all comparison stars.
  • The vertical scale is counts in each bin, with values of 10 or greater being at the top of the plot.
  • In each plot, the largest modified distance for that star is plotted in Red.
  • For the target star, if the 2nd, or 2nd +3rd, modified distances are consecutive with the largest modified distance, the modified distances for the 2nd, or 2nd + 3rd, are plotted in Yellow
  • Lists of the 20 largest modified distances, together with their frame number, is provided. for the target and all comparison stars. This allows for the user to readily access the numerical values of the modified distances and make a manual interpretation. Each list is generated specifically for that star, and generally has no correlation with any other list. The data in each list can be copied by placing the cursor over the box, and right-click.
  • The list for the target star also shows the modified distances for each Comparison star at the same frame.
  • Above each plot there is a number giving the relative brightness of the comparison stars relative to the Target star. Desirably, the comparison stars should be of similar brightness to the target. Relative brightness values greater than 5 times the brightness of the target,  or less than 0.2 times the target, are indicated in dark red, with the list of distance values in pink. This is to indicate that the values for this star may not be reliable.
  • At the left of the panel is a box for the evaluation of the data. The evaluation is whether an occultation event is either Probable, or Possible - with an occultation event otherwise being unlikely. The relevant frame # is given.

The resulting Scaling factor is generated with

Assessment

The assessment of whether an occultation event occurred is not a black and white test. It is possible (but not a routine occurrence) that there are additional characteristics or features of a particular light curve that might outweigh the following criteria, which will need to be considered in the context of that particular light curve.

The issues to consider are:

  1. The primary indicator is whether the maximum value in the target box is more than the scaled ratio value of the 2nd value.
    The simple visual test for this is: in the graph for the Target star, is there a large gap between the data plotted in light blue, and the data point in Red - at the right side of the plot.
    For two (& 3) frame events, the height of an adjacent frame would be expected to be similar to that of the primary frame. Such frames are plotted in yellow, to indicate they are excluded from the test. That is, the scaled ratio test is applied between the 1st and 3rd (or 4th) values.
  2. The light drop for the target star must be more than scaled ratio value of the largest value in all other light curve drops. When the largest light drop in the target star has a similar (or greater) light drop in any of the companion stars, you have a clear indication that the light drop in the target star is noise, not an occultation.
    The simple visual test for this is: Is there a gap between the the red mark in the graph for the target star, and the right-hand end of that graph. A gap means there is a greater light drop in one of the comparison stars.
    However this interpretation may not be correct if the relevant companion star is much brighter or fainter than the target star. That is, when the relevant list is pink. Greater regard should be had to the other comparison stars.
  3. The distribution of the data displayed in the target and comparison stars should be similar, as an indication that the noise signals are similar in nature. A major difference indicates that conclusions relying on that comparison star may be unreliable.
  4. If there are no comparison stars, any confidence that a single (or short-term) event was not caused by noise is reduced compared to when the noise of one or more comparison stars are available. Hence the large ratio value when there are no comparison stars.

Automatic interpretation

The evaluation box gives the numeric value of the primary indicator. It is color coded as follows:

  • Values above the scaled Probable Occultation ratio: Green, with a tick. Such events can be considered for reporting astrometry (subject to any other considerations that might arise with the overall event processing)
  • Values above the scaled Possible Occultation ratio: Blue, with a question mark. An event which could plausibly have been caused by an occultation, but not sufficient evidence to report astrometry. Such events can be included in the file of observations, with the Quality of the Fit code set to 'Short duration event. No astrometry'.
  • When the greatest value for the target star is less than the scaled Possible Occultation ratio of any star: Displayed in Orange, with ??. This will typically indicate the event is noise. However this may not be a reliable conclusion if the largest value is in a pink box. The relationship to the largest value in other comparison stars should be considered.
  • Values less than the scaled Possible Occultation ratio: Red, with a X . This indicates the largest excursion is indistinguishable from the noise. Such an event will not be added to the file of observations unless the observer provides the reason they believe there is an occultation event, which will be included as part of their observation of the event. Such reasons should be less than 100 words, and expressed in plain ASCII text with no paragraph breaks.
  • When the two or three largest values for the target star are in adjacent frames, the numeric value is followed by  *  , with that numeric value being the ratio of the 1st and 3rd (or 4th) values.

 

Single chord events - coincidence with a prediction

The predicted time of an event is, without doubt, relevant to assessing where in the recording one should start looking for an event. The length of the light curve to be considered is typically several ten's of seconds. For slow-moving asteroids, it may be minutes.

This is the ONLY purpose for using the predicted event time. The conclusion of whether a particular dip in a light curve is caused by an occultation must stand on its own. Coincidence, or near-coincidence, with the predicted time, is irrelevant and does not constitute a basis for asserting the observation was an occultation.

There are those that want to use a Bayesan argument to give an event a higher probability if it occurs near the predicted time. That argument assumes a 'prior' (all Bayesian calculations depend on the 'priors' used) that is not flat - that is, we know the probability distribution of the event time. However we do not know that distribution. Bayesians reflect this lack of knowledge by using a flat prior (that is, the event has an equal probability of occurring at any time), and as a result the Bayesian calculation is unaffected by the observed event time.

 

Multichord events - interpretation

The concerns raised with Single chord events apply to multichord events.

The additional factor is coincidence with other observed chords. If the other chords are multi-frame observations, and you have a single frame observation, the possible interpretations are:

  1. the single frame chord is close to the multiframe chords, such that from those multiframe chords we know an occultation would have occurred. As a result, the single frame result probably will add nothing to the multichord results. If it is included in the solution, the effect might reduce the accuracy/reliability of the overall result. In such cases, the event will be included in the file, but excluded from analysis.
  2. the single frame chord is remote from the multiframe chords, such that the possibility of there being an occultation is remote (at best). The inference would be that the event that was considered to be a possible occultation was a noise event. Such an event would be included as reported, but excluded from any analysis.
  3. The single frame chord is arguably near where the limb of the asteroid profile might pass, such that it could be considered a positive event that assists in determining the size/shape/position of the asteroid. However such a conclusion is dependent upon the level of certainty about whether the recording was caused by an occultation, because the same consequences arise when the chord is treated as a Miss chord.

 

Light curve - depth of drop

This tool does not include any consideration of whether the observed light drop is consistent with the predicted drop. For asteroids larger than 1 or 2 km, there should be consistency between the predicted and observed light drops. If the observed light drop is significantly less than predicted, it is unlikely to be caused by an occultation.

For small asteroids, the effects of Fresnel diffraction need to be considered.

3-frame events - is the fact there are 3 consecutive points below the average light level sufficient to assert it was an occultation?

Maybe, maybe not. The distinguishing feature of an occultation is a light curve with a flat bottom. Apart from the particularly wide event in the middle of he plot, there are two other drops extending over 3 frames, each of which might be construed as an occultation if it was the only such light drop in the light curve. As a general proposition, such events should be considered under this test; however the presence of three points at a similar light level may justify not strictly applying the evaluation. However note how in this example the 'flattest three frame drop (leftmost) is, in the context of this light curve, definitely not an occultation event.
